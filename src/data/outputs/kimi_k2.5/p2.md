# API通信を料理に例えて説明

## 結論：レストランでの注文と同じ構造

API通信は、**お客様（アプリ）がレストラン（サーバー）に料理（データ）を注文する**仕組みと同じです。

---

## 各要素の対応関係

| API用語 | 料理の例 | 説明 |
|---------|---------|------|
| **クライアント** | お客様 | 注文する人（アプリやブラウザ） |
| **サーバー** | レストラン・厨房 | 料理を作る場所 |
| **リクエスト** | 注文票 | 「これをください」と伝えること |
| **レスポンス** | 出来上がった料理 | 注文したものが届く |
| **エンドポイント** | レジ・窓口 | 注文を受け付ける場所 |
| **APIキー** | 会員証・予約番号 | 「正規のお客様です」証明 |

---

## 具体的な流れ

### 1️⃣ メニューを見る = APIドキュメントを確認
```
お客様「メニュー（仕様書）をください」
↓
レストラン「はい、これが提供できる料理（データ）の一覧です」
```

### 2️⃣ 注文する = リクエスト送信
```
お客様「ハンバーグ定食（ユーザー情報取得API）をください。
        味噌汁は抜きで（パラメータ指定）」
↓
【注文票（リクエスト）】
・料理名（エンドポイント）: /api/users/info
・オプション（パラメータ）: { miso_soup: false }
・会員番号（APIキー）: XXXX-XXXX
```

### 3️⃣ 厨房で調理 = サーバー処理
```
厨房「注文受けました！ハンバーグを調理します」
（データベースから情報を取り出して準備中...）
```

### 4️⃣ 料理が届く = レスポンス受信
```
【出来上がった料理（レスポンス）】
{
  "status": "success",
  "data": {
    "main_dish": "ハンバーグ",
    "rice": "大盛り",
    "soup": null,
    "cooking_time": "15分"
  }
}
```

### 5️⃣ 会計 = 利用制限（レートリミット）
```
店員「お客様、1時間で5食までの注文制限があります。
      次の注文は15分後にお願いします」
```

## HTTPメソッドを料理に例えて

| HTTPメソッド | 料理での行動 | 具体例 |
|-------------|-------------|--------|
| **GET** | 見る・確認する | メニューを見る、料理の写真を見る |
| **POST** | 新規注文 | 新しい料理を注文する |
| **PUT** | 内容を変更 | 「ハンバーグをチキンに変更」 |
| **DELETE** | キャンセル | 「注文を取り消し」 |
| **PATCH** | 一部変更 | 「味噌汁だけ大盛りに」 |

---

## エラーの例え

| ステータスコード | 料理での状況 | 意味 |
|-----------------|-------------|------|
| **200** | 「はい、お待ち！」 | 正常に提供完了 |
| **400** | 「メニューにない注文です」 | リクエストが変 |
| **401** | 「会員証を見せてください」 | 認証エラー |
| **403** | 「その料理は注文できません」 | アクセス権限なし |
| **404** | 「その料理は売り切れです」 | 見つからない |
| **500** | 「厨房で火事が...」 | サーバー内部エラー |
| **503** | 「満席です、後で来て」 | サービス利用不可 |

## 実際のAPI通信イメージ

### 天気予報APIを使う場合

```
【注文（リクエスト）】
店：天気レストラン
注文：「明日の東京の天気をください」
↓
【提供（レスポンス）】
{
  "location": "東京",
  "date": "明日",
  "weather": "晴れ",
  "temperature": "25度"
}
```

### 決済APIを使う場合

```
【注文（リクエスト）】
店：銀行レストラン
注文：「AさんからBさんに1000円振り込んで」
↓
【提供（レスポンス）】
{
  "status": "完了",
  "from": "Aさん",
  "to": "Bさん",
  "amount": 1000,
  "balance": 5000
}
```

---

## まとめ

> **API通信 = レストランで料理を注文する仕組み**

- **クライアント**（お客様）が **サーバー**（レストラン）に
- **リクエスト**（注文）を送ると
- **レスポンス**（料理）が返ってくる

この「注文・提供」の決まり事（ルール）が **API（Application Programming Interface）** です。お客様と厨房が直接話さなくても、レジ（API）を通じてスムーズにやり取りできるのが便利なところです。
